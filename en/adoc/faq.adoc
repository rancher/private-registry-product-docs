[#product-overview-and-differentiators]
=== Product overview and differentiators

[.qanda]
What kind of subscription do customers need for {productname}?::
It is included in {ranchera} Suite and offered as an add-on for {ranchera} Prime.
+
The pricing of the add-on is the same as other add-ons.

What are the differentiators with {harbor} (from Application Collection or upstream)?::
Use {productname} if you need:
* L3 support for product issues.
* A predictable release cycle.
* Patched images for known vulnerabilities. Upstream {harbor} images from {dhub} often contain link:https://artifacthub.io/packages/helm/harbor/harbor?modal=security-report[numerous unpatched vulnerabilities].
+
Over time, {suse} will also add out-of-the-box integrations with {ranchera} and prioritize feature requests from customers.

Do customers have to buy the same number of add-on {productname} subscriptions as {ranchera} Prime subscriptions?::
Yes, just like other add-ons such as {suse} Security. An additional advantage of the {productname} subscription model is that it allows customers to run as many deployments as they need.

[#relationship-with-upstream-harbor]
=== Relationship with upstream {harbor}

[.qanda]
How does the release cycle align with upstream {harbor}?::
The release cycle of {productname} is independent of the release cycle of upstream {harbor}.
+
When releasing new versions of {productname}, {suse} aims to include the latest version of the upstream {harbor} project that meets SUSE quality assurance and maintenance requirements.

Will you publish migration considerations for customers running {harbor} from other sources?::
Not at this time.

[#deployment-and-installation]
=== Deployment and installation

[.qanda]
Do you support installing {productname} via {compose}?::
No. You must install and configure {productname} using its {helm} chart. This chart is also used for ongoing management (Day 2 operations) and can be integrated with GitOps workflows.

Do you recommend deployments on the local ({ranchera} Manager) cluster?::
No. SUSE recommends deploying {productname} on a downstream cluster. This makes the registry accessible to other downstream clusters that need to consume images.

What are the deployment best practices for a high-availability (HA) environment?::
For HA, do not deploy {productname} on the {ranchera} Management cluster to avoid resource contention.
+
You can deploy {productname} on a dedicated cluster, or on one or more of your application clusters.
+
For instructions, see xref:pr-deployment-ha[]. Note that you must provide your own HA components for the Postgres database, Valkey/Redis server, and {ingress} controller. These components are not deployed by the {productname} {helm} chart and are not supported by {suse}.

{productname} does not ship with a dedicated operator. It is installed and managed using its {helm} chart, which can be integrated with GitOps. {suse} continues to evaluate operator-based management for future releases.

[#security-scanning-and-signing]
=== Security, scanning and signing

[.qanda]
Do you integrate any signing tool within {productname}?::
{productname} does not sign images itself, but it can store, distribute and verify OCI-compatible signatures.
+
A widely used tool is `cosign`, which can sign images and store the signatures in {productname}.
+
`cosign` signatures attached to images are viewable and downloadable from the {productname} portal.

Is it possible to sign images with Notary or Cosign for approval during the deployment process?::
{productname} can store, distribute and verify `cosign` signatures.
+
{productname} itself does not sign images. You must sign images during their build process, and then upload the signature to the registry along with the image.
+
Notary is not included or supported with {productname}.

Are images scanned for CVEs only with Trivy, or can it use {suse} Security/NeuVector as well?::
Images are scanned with Trivy, which comes out of the box. {suse} Security can be added in addition to or as a replacement for Trivy.

Is it possible to run ClamAV or similar malware scans?::
By default, {productname} scans images using Trivy. You can also configure {suse} Security (NeuVector) as a scanner.

In the pull-through cache scenario, will the vulnerability criteria be enforced?::
Yes, but with a known limitation. Due to upstream {harbor} link:https://github.com/goharbor/harbor/issues/16218[issues], vulnerability criteria are not enforced on the first pull of an image because the scan has not yet completed.
+
For complete coverage, pair {productname} with {suse} Security link:https://open-docs.neuvector.com/policy/admission#admission-control-criteria[admission controls].

Are the images of {productname} hardened?::
Yes. The container images for {productname} are hardened. They are based on SLE BCI and built in the same enterprise-grade SUSE Build Service used for SUSE Linux Enterprise. This ensures a secure supply chain. The images are also signed, and SUSE publishes their SLSA attestations.

Are the images of {productname} signed? How?::
The images are signed with `cosign`. You can verify them by saving the PEM-formatted signing key published at link:https://www.suse.com/support/kb/doc/?id=000021411[KB 000021411].
+
Then run, for example:
+
[source,bash]
----
cosign verify --key container-key.pem registry.suse.com/private-registry/harbor-portal:latest
----

[#features-and-integration-with-ranchera]
=== Features and integration with {ranchera}

[.qanda]
Is there a plan to integrate the Registry within {ranchera} using an Extension?::
Yes. {suse} plans deeper integration with {ranchera} Prime and other offerings. Future enhancements being considered include SSO integration, simplified setup for {ssecurity} scanners and Application Collection mirroring, monitoring with {sobservability}, and a {ranchera} UI extension.

Will {suse} add the Registry to the Training Catalog?::
This is not yet planned. If you are interested in training material, contact {suse} to discuss possibilities with the Training team.

[#support-and-documentation]
=== Support and documentation

[.qanda]
Is there a support policy for {productname}?::
Yes. The support policy of {productname} is the same as any other {ranchera} Prime add-on.

If the {productname} documentation lacks information, can I refer to the official {harbor} documentation?::
Yes. Since {productname} is based on {harbor}, the official {harbor} documentation is a useful resource. For features specific to the SUSE version, refer to the {productname} documentation.
+
The xref:pr-relnotes[] specify which upstream {harbor} version corresponds to your {productname} version.
